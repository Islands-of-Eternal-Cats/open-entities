### С какими проблемами производительности вы столкнетесь?

Поскольку вы интересовались скоростью рефлексии, при работе с JS нужно учитывать три главных «узких места».

При создании системы модинга на JavaScript в Bevy вы столкнетесь с тремя основными bottlenecks, которые могут сильно уронить FPS игры.

---

## 1. Маршалинг данных (Boundary Crossing)

**В чём суть:** память Rust (где лежат компоненты ECS) и память JavaScript-рантайма — это два абсолютно разных мира. JS не может напрямую прочитать память структуры Rust.

**Как это работает:** каждый раз, когда мод пишет `player.hp = 100`, происходит тяжёлая цепочка действий. Сначала `bevy_reflect` ищет поле `hp` по строковому имени, затем рантайм конвертирует число `f32` из Rust в JS-совместимый тип данных (выделяя под него память в куче JS), и только потом скрипт производит вычисление.

**Почему это тормозит:** если 500 мобов на сцене будут каждый кадр запрашивать свои координаты `x`, `y`, `z` через JS, игра превратится в слайд-шоу из-за постоянной конвертации типов туда-обратно.

---

## 2. Сборщик мусора JavaScript (Garbage Collection / GC)

**В чём суть:** Rust управляет памятью во время компиляции (через правила владения) и работает невероятно быстро. У JavaScript своя философия — он выделяет память в процессе работы, а затем периодически запускает сборщик мусора (GC), чтобы удалить ненужное.

**Как это работает:** если JS-мод внутри игрового цикла (каждый кадр) создаёт временные объекты, массивы или строки (например, для расчёта траектории или вывода текста), память под них забивается мгновенно.

**Почему это тормозит:** рано или поздно рантайм JS поставит выполнение скрипта «на паузу» (пусть и на несколько миллисекунд), чтобы очистить мусор. Для игрока это будет выглядеть как внезапный микрофриз или раздражающее заикание картинки (stuttering).

---

## 3. Потеря многопоточности (Single-Threaded JS Bottleneck)

**В чём суть:** Bevy прекрасен тем, что его ECS автоматически распределяет системы по всем доступным ядрам процессора (параллельность из коробки). Однако стандартный JavaScript по своей природе однопоточен.

**Как это работает:** когда запускается JS-рантайм, он обычно работает в одном конкретном потоке.

**Почему это тормозит:** если тяжёлый JS-мод начнёт обсчитывать сложную логику, он намертво заблокирует тот поток Rust, в котором он был вызван. Пока JS-скрипт не завершит работу, Bevy не сможет продолжить кадр. В итоге мощный 8-ядерный процессор игрока будет простаивать, пока одно ядро захлёбывается в JS-коде мода.

---

## Как это лечат в реальных играх?

Чтобы обойти эти ограничения, разработчики используют архитектуру **«Толстый Rust, тонкий JS»**:

- **Событийная модель (Event-driven):** запретите JS-скриптам работать каждый кадр (`update`). Вместо этого вызывайте JS только по событиям: `on_weapon_fired()`, `on_player_death()`.
- **Пакетная передача (Batching):** вместо того чтобы гонять координаты каждого врага по отдельности, Rust-система может раз в несколько кадров собрать нужные данные в один компактный массив и передать его в JS за один раз.
